# 05 测试与交付工程师任务书（15%）

## 目标

建立独立于开发者自测的验收体系，记录真实缺陷，确认 Docker 交付从 README 可复现，并形成老师可以直接检查的测试、截图、Bug 和贡献资料。

## 前置阅读

1. 原始 Day08 任务书
2. `README.md`
3. `docs/PRD.md`
4. `docs/API.md`
5. `docs/GIT_WORKFLOW.md`
6. 四名成员 PR

从包含 CV、后端和前端最新成果的 `main` 创建 `feature/qa-delivery`。

## 独占路径

- `tests/fixtures/`
- `tests/test_acceptance.py`
- `tests/test_security.py`
- `tests/test_delivery.py`
- `scripts/package_release.py`
- `docs/TEST_REPORT.md`
- `docs/BUG_RECORD.md`
- `docs/ACCEPTANCE_CHECKLIST.md`
- `docs/CONTRIBUTIONS.md`
- 最终验收截图索引

发现功能 Bug 时先写失败测试和 Bug 记录，再让对应文件责任人修复；QA 不直接接管其他成员模块。

## 测试数据

准备并注明来源：

- 一张无 `enemy` 的通过图片；
- 一张中等置信度风险的待复核图片或可控假检测验收夹具；
- 一张高置信度风险的不通过图片；
- 一个 5–15 秒合法 H.264 视频；
- 一个损坏图片；
- 一个损坏视频；
- 一个零字节文件；
- 一个错误扩展名文本文件。

真实模型验收必须使用真实素材；规则边界可以使用注入式假 Detector 保证确定性。

## 正常测试不少于 5 条

1. 图片创建、分析、完成和报告；
2. 图片三档审核规则；
3. 视频异步状态和证据帧；
4. 人工改判、负责人和备注；
5. 刷新后历史任务重开；
6. JSON/CSV/ZIP 下载；
7. 完成任务删除；
8. 统计随最终结论更新。

## 异常测试不少于 5 条

1. 不支持扩展名；
2. 空文件；
3. 损坏媒体；
4. 模型文件不存在；
5. pipeline 抛出异常并落盘失败状态；
6. 运行中删除；
7. 重复分析；
8. 非法任务 ID；
9. `../` 产物路径；
10. 阈值顺序错误；
11. 负责人为空；
12. 服务重启恢复遗留运行任务。

## 真实 Bug 流程

至少记录两个实际出现的问题：

1. 保存复现步骤、请求或截图；
2. 写出失败测试；
3. 分析原因，不能只写“代码错误”；
4. 指派原模块成员修复；
5. 合入后用相同步骤和自动测试复验；
6. 在 `BUG_RECORD.md` 记录修复提交和结果。

禁止先编造两个 Bug 再倒推修复记录。

## 交付检查

- Docker 构建和启动；
- Conda 备用启动；
- `/api/health` readiness；
- 全部自动测试；
- `node --check`；
- 真实模型烟雾测试；
- JSON 可以重新解析；
- CSV 列名与报告一致；
- ZIP 可列出并 CRC 校验；
- 页面截图和结果截图；
- 无密钥、绝对隐私路径、缓存和无关大文件；
- 五人贡献映射；
- 8 分钟演示稿；
- 最终目录命名 `<组长姓氏>_A_day08`。

## 建议提交

1. `test: add normal and abnormal acceptance matrix`
2. `test: cover path safety recovery and delivery artifacts`
3. `docs: record verified bugs and regression evidence`
4. `build: add deterministic release package validation`

## 验证

```bash
pytest -q
python -m py_compile app.py
node --check static/app.js
docker compose build
docker compose run --rm app pytest -q
docker compose up -d
curl --fail http://127.0.0.1:7880/api/health
unzip -t <最终审核包.zip>
git diff --check
git shortlog -sne --all
```

## 验收标准

- 5 条正常、5 条异常均有输入、预期、实际和证据；
- 两个真实 Bug 有失败测试、原因、修复提交和复验；
- README 从干净容器可运行；
- 至少一个真实图片和一个真实视频结果；
- 输出文件可打开、CSV 可读、ZIP 完整；
- 五名成员均有能现场讲解的独立产出。

## 停止条件

Docker 无法启动、模型缺失、核心流程没有真实推理、测试存在失败或真实成员信息未补齐时，不得生成“最终通过”报告或最终提交包。
